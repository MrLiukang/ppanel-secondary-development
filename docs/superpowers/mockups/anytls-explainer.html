AnyTLS：包到底有什么不同？

先给结论：AnyTLS 不会把 TCP 变成一种新的物理传输。在线上，你看到的仍主要是 TCP + TLS；不同点在于 TLS 加密之后，应用数据如何被组织、分片，以及是否加入可配置的 padding。

“看不到明文” ≠ “看不到流量特征”

抓包通常看不到应用内容，但仍能看到 IP、端口、TCP 状态、TLS 握手元数据、记录长度、方向和时间间隔。AnyTLS 的 padding/分片会改变部分长度与时序形状，但不等于完全不可识别。

它究竟位于哪一层？

把“协议名”拆开看：AnyTLS 负责会话里的数据组织；TLS 负责加密；TCP 负责可靠传输。

应用数据→AnyTLS 会话→TLS 加密→TCP→IP
点击一层看它负责什么
应用数据：这是最里面的明文语义。经过 TLS 后，旁路抓包者通常不能直接读取它；服务器端解密后才交给上层处理。

它是不是 HTTP 伪装？

不是。AnyTLS 不等于把流量包装成可读的 HTTP 请求；更准确地说，外层仍是 TLS 加密连接，HTTP（如果有）只是被保护的应用内容之一。

包里会出现 “AnyTLS” 吗？

通常不会出现一个明文的“AnyTLS”包头供旁路直接匹配。协议识别仍可能依赖握手特征、实现差异、连接行为或统计形状。

映射到当前项目的配置

项目中的 buildAnyTLS 将入站协议设为 anytls，传输网络设为 tcp，并把换行分隔的 padding 规则传给 Xray 的 AnyTLS 服务端配置。

概念配置

protocol: anytls
streamSettings:
  network: tcp
settings:
  paddingScheme:
    - "stop=8"
    - "0=30-30"
    - "1=100-400"
    - "2=400-500,c,500-1000,c,..."

项目代码对应关系

// node/core/inbound/anytls.go
inbound.Protocol = "anytls"
t := coreConf.TransportProtocol("tcp")
inbound.StreamSetting = &coreConf.StreamConfig{
  Network: &t,
}
settings := &coreConf.AnyTLSServerConfig{
  PaddingScheme: padding,
}
protocolanytls 选择 AnyTLS 入站处理逻辑。
networktcp 表示底层是 TCP 字节流，不是 UDP/QUIC。
paddingScheme每行是一条填充规则；它影响发送形状，不会让明文内容变得可见。
验证边界配置能证明“服务端如何启动”，不能单独证明某次客户端抓包一定采用了某个具体规则。
三个最容易混淆的结论

不是新底层传输

当前项目明确把 AnyTLS 绑定到 TCP；观察到的底层连接仍是 TCP。

不是绝对隐身

TLS 隐藏内容，但 IP、端口、握手、记录长度和时序仍可能被观察。

padding 不是加密

padding 主要改变长度/节奏等外观；机密性仍由 TLS 提供。
