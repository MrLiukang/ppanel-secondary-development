# AnyTLS 协议解释器 HTML 设计

## 目标

制作一个可离线打开的单文件 HTML 页面，用中文同时服务三类读者：入门用户、运维排障人员、节点开发者。页面需要回答 AnyTLS 是什么、它在线上发送的包是否与普通 TLS 有区别，以及当前项目配置如何映射到协议行为。

## 核心判断

页面必须区分三个层次：

1. AnyTLS 的会话/封装语义。
2. TLS 加密后在线路上可观察到的记录、长度和时序。
3. TCP/IP 的连接与传输特征。

“应用内容不可见”不等于“流量完全没有特征”。对无法仅凭抓包确认的结论，使用“通常可见 / 通常不可见 / 只能推测”的表述。

## 页面结构

- 顶部结论卡：AnyTLS 的一句话定义、是否使用 TCP、是否等同 HTTP 伪装。
- 入门标签：展示 `应用数据 → AnyTLS 会话层 → TLS 加密 → TCP → IP` 分层关系，点击层级显示职责。
- 抓包标签：并排比较普通 TLS、TLS 内 HTTP/2、AnyTLS 的可见字段、不可见内容与长度/时序特征。
- 时序图：模拟 TCP 建连、TLS 握手、加密应用数据、分片与 padding 的先后关系，并提供数据包点击说明。
- 配置标签：展示 `protocol: anytls`、`network: tcp`、`paddingScheme`，以及项目中 `buildAnyTLS` 的对应关系。
- 误区卡片：说明“包里不会直接出现 AnyTLS 字样”“不是 HTTP 伪装”“不能承诺完全不可识别”等结论。

## 交互与视觉

- 使用单页标签切换，不引入外部库或网络资源。
- 使用深色技术文档风格，以蓝绿色表示已加密/安全边界，以橙色表示可观察但不等于明文的元数据。
- 数据包、协议层和配置字段均可点击，点击后在同一页面展开解释。
- 响应式布局，窄屏下卡片改为单列。

## 验收标准

- 双击 HTML 文件即可打开，且无外部依赖。
- 普通读者能看懂 AnyTLS 与 TLS、TCP 的关系。
- 运维读者能看到握手、记录长度、方向和时序等抓包观察点。
- 开发者能看到当前项目 `node/core/inbound/anytls.go` 中 TCP 与 padding 配置的映射。
- 页面明确标注：TLS 加密保护内容，但不消除连接元数据与流量形状。
- 页面脚本无控制台错误，主要交互在桌面和移动宽度下均可用。
